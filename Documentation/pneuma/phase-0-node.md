# Pneuma Phase 0 — Adopt the Omarchy VM as GPU worker `pneuma`

Executed 2026-07-02. The existing Omarchy gaming VM (VMID 100 on furina) was
adopted as the cluster's single GPU worker — **one VM for gaming + inference**,
arbitrated with `scripts/game-mode.sh` / `scripts/work-mode.sh` instead of VM
swapping. No Terraform, no PVE-cluster join (furina stays standalone).

## What was done (re-runnable)

Access path: the VM's SSH is Tailscale-native (interactive check) — automation
uses **`qm guest exec` from furina** (`ssh root@192.168.1.103`) instead.

1. Rollback anchor: `qm snapshot 100 pre-pneuma` (disk+efi+tpmstate, guest FS frozen).
2. In-VM prep (`/root/pneuma-prep.sh`, kept on the VM):
   - `hostnamectl set-hostname pneuma` + `/etc/hosts` entry
   - static IP file (sorts before the stock `20-ethernet.network` DHCP match)
   - `mkdir -p /var/lib/vllm-models /etc/k0s/containerd.d`
   - containerd drop-in `/etc/k0s/containerd.d/nvidia.toml` (nvidia runtime →
     `/usr/bin/nvidia-container-runtime`; k0s watches this dir)
   - `pacman -Syu --noconfirm nvidia-container-toolkit` (full upgrade — system
     was already current, only toolkit 1.19.1 added; CDI regenerated by hook)
   - k0s binary pinned: `curl -sSLf https://get.k0s.sh | K0S_VERSION=v1.35.2+k0s.0 sh`
3. `qm set 100 --memory 49152 --onboot 1` + graceful restart — validated
   unattended boot (LUKS keyfile auto-unlock, autologin, static IP applied).
4. Join (raiden-rebuild precedent — token join, never `k0sctl apply` at a live cluster):
   - controller: `sudo k0s token create --role=worker --expiry=1h`
   - pneuma: `k0s install worker --token-file /root/join-token
     --labels "teyvat.io/gpu=true" --taints "nvidia.com/gpu=present:NoSchedule"
     && k0s start`

Notes: inotify limits were already sane on Arch (1024 > the 512 we require on
Ubuntu nodes — no sysctl change needed). The desktop session idles ~0.2–0.6GB
VRAM; vLLM runs `--gpu-memory-utilization=0.85` to leave room for it.

## Game-mode / work-mode

- `scripts/game-mode.sh` — cordon pneuma + delete pods labeled
  `teyvat.io/gpu-workload=true`. VRAM frees in seconds; node/Sunshine stay up.
  LLM requests queue or 504 until work-mode.
- `scripts/work-mode.sh` — uncordon; KEDA cold-starts vLLM on the next request.
- Never `kubectl scale` the vLLM deployments (KEDA owns replicas; ArgoCD
  self-heal reverts drift).

## Arch update procedure (risk #1 for this node)

1. `scripts/game-mode.sh` (nothing on the GPU), then on furina:
   `qm snapshot 100 pre-update`
2. In the VM: **full** `pacman -Syu` only — never partial (`-Sy pkg`); kernel
   and nvidia-open must move together.
3. Reboot. Verify `nvidia-smi`, node Ready, gpu-operator validator pods green.
4. `scripts/work-mode.sh`. Delete old snapshots once happy
   (`qm delsnapshot 100 pre-update`).

k0s itself is NOT pacman-managed (pinned static binary at `/usr/local/bin/k0s`).
Upgrade it only deliberately, matching the cluster version.

## Teardown

```
# on pneuma:  k0s stop && k0s reset
kubectl delete node pneuma
# VM reverts to a pure gaming box; optional: qm rollback 100 pre-pneuma
```

The `pre-pneuma` snapshot on VM 100 can be deleted once the platform has been
stable for a while (`qm delsnapshot 100 pre-pneuma`).
